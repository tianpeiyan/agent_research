# 开发指南

本文档固化当前编排、业务工具和外部 skill 的开发约定。

## 放在哪里

新增能力先按职责判断边界：

- LangGraph node：负责流程状态推进、条件分支、循环上限和失败兜底。
- 普通服务函数：负责可复用的确定性业务逻辑，适合被 node 或工具调用。
- ToolRegistry tool：负责 agent 可选择的业务动作，必须有参数 schema、统一日志、错误码和脱敏。
- Agent：负责 LLM prompt、结构化输出解析和 agent 级重试。
- Skill：只提供只读操作手册，不执行业务动作。

如果能力需要被 LLM 自主选择，优先放入 `ToolRegistry`。如果能力只决定流程下一步，放入 orchestrator node 或普通 route/helper。不要在 agent prompt、ReportWriter 或 skill 中绕过 registry 执行搜索、写文件或外部命令。

## 新增 LangGraph Node

当前图由 `ResearchOrchestrator._build_graph()` 构建，节点使用动词短语命名，例如 `judge_evidence`、`rewrite_queries`。

新增 node 时：

- 使用 `ResearchWorkflowState` 作为输入和返回类型。
- 只返回该 node 修改的 state 字段。
- route 函数命名为 `route_after_<node>`。
- 所有循环 route 必须检查上限字段。
- node 内的业务动作通过 `ToolRegistry.execute()` 调用。
- 非关键节点失败时应写入可诊断状态并继续生成报告。

图结构变更后，更新 `tests/backend/test_orchestrator.py` 中的 graph 节点/边测试。

## 新增业务工具

业务工具集中注册在 `ToolRegistry`。

新增工具时：

- 增加 Pydantic args model，`extra="forbid"`。
- 注册 name、description、args model、handler、stage。
- handler 只读写 `ToolExecutionContext` 中明确字段。
- 返回可 JSON 序列化 payload。
- 失败使用 `ToolExecutionError(code, message)` 或项目内 `ToolError`。
- 在 `_input_summary()` 和 `_output_summary()` 中增加简短日志摘要。
- 补测试：参数校验、成功路径、失败错误码、工具日志和上下文更新。

所有 agent 默认共享完整 business tool definitions；需要限制某个 agent 的可用工具时，使用 runner 的 `allowed_actions`。

## 新增外部 Skill

skill 放在项目根目录：

```text
skills/<skill_name>/SKILL.md
```

第一版限制：

- `skill_name` 只能包含字母、数字、下划线和连字符。
- 只读取 `SKILL.md`。
- 不执行 `scripts/`。
- 不读取 skill 目录外文件。

安装或新增 skill 后，agent 只能通过 `load_skill` 工具读取手册。测试应覆盖正常读取、缺失 skill、缺失 `SKILL.md` 和路径穿越拒绝。

## 调试流程状态

编排状态字段定义在 `backend/app/research_state.py`：

- `planned_tasks`
- `current_index`
- `execution_records`
- `completed_summaries`
- `failed_tasks`
- `sources`
- `evidence_judgement`
- `supplemental_queries`
- `supplemental_rounds_used`
- `report`

调试时优先看：

- `ResearchResult.tool_logs`：每个工具调用的 stage、tool_name、输入摘要、输出摘要、状态和脱敏错误。
- SSE `status` / `task` / `search_results` / `summary` / `report` / `done` 事件。
- graph 编译结构：`orchestrator._build_graph(executor).get_graph()`。

不要把 API key、Bearer token、secret、token 等敏感信息写入日志、SSE 或用户可见错误；统一通过 `ResearchProgressTracker.sanitize()` 处理。

## 报告与证据状态

`ReportWriter` 只负责写报告和引用校验，不负责搜索调度。

当前报告输入包括：

- topic
- completed summaries
- evidence judgement

如果证据不足，ReportWriter prompt 必须要求标注风险、区分高置信结论和待验证结论，并避免夸大证据。fallback report 也必须包含 evidence 状态。补充搜索只由 orchestrator 的 `rewrite_queries` / `supplemental_search` 流程触发。
