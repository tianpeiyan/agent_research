# 项目级开发规范

## 开发主导权

本项目由 Codex 主导日常开发实现。用户负责提出目标、业务约束和验收方向；Codex 负责技术方案、框架取舍、代码风格、模块边界、测试策略和实现细节。

当用户没有明确指定实现方式时，Codex 应自主选择最利于代码质量、可维护性、可扩展性和交付稳定性的方案，不需要为常规技术细节反复询问。

## 最高目标

所有开发决策优先服务于以下目标：

- 代码质量高：逻辑清晰、边界明确、错误可诊断、测试可覆盖。
- 可维护：模块职责稳定，避免隐式耦合，避免临时补丁。
- 可扩展：新增 agent、工具、流程节点或外部能力时，不需要重写主流程。
- 可测试：关键分支、失败路径、安全约束和预算限制都必须能被自动化测试验证。
- 可观测：长流程、工具调用、失败兜底和敏感信息脱敏应有明确日志或事件。

## 技术选型原则

Codex 拥有项目技术选型决策权，但必须遵守以下判断标准：

- 框架只在明显降低复杂度、提高可靠性或减少自研风险时使用。
- 业务强约束、安全边界、错误码、预算、日志、脱敏、引用校验和 skill 沙箱边界优先由项目代码明确实现。
- 当前已使用的 LangGraph 可继续作为编排基线，但后续开发不强制绑定 LangGraph 或任何特定框架。
- 如果框架抽象使实现更绕、测试更脆弱或调试更困难，应使用清晰的普通服务函数或项目内抽象。
- 不为未来假设过度设计；只有当重复、复杂度或扩展需求已经明确时才抽象。

## 代码风格

Codex 决定代码风格，但必须保持全仓一致并贴合现有代码：

- Python 代码使用类型标注，公共边界优先使用 Pydantic 模型或明确的 TypedDict/dataclass。
- 异步代码只在外部 IO、LLM、工具调用、搜索、文件或网络操作需要时使用。
- 函数应短小、职责单一；复杂流程拆成清晰的私有 helper 或服务对象。
- 错误必须携带可测试错误码或可诊断信息，不能只返回模糊字符串。
- 敏感信息不得进入用户可见错误、日志、SSE 事件或报告。
- 不引入无关格式化、重命名或大规模重构。
- 注释只解释非显然的设计意图或安全边界，不重复代码表面含义。

## 架构边界

默认架构边界如下：

- Orchestrator / 编排层：负责流程调度、状态推进、条件分支和失败兜底。
- Agent 层：负责 LLM prompt、结构化输出解析和 agent 级重试。
- ToolRegistry：负责业务工具注册、参数校验、白名单执行、工具日志和脱敏。
- SkillRegistry：负责外部 skill 发现和只读安全加载。
- ReportWriter：负责报告生成、证据状态表达和引用校验，不负责搜索调度。
- ProgressTracker：负责 SSE 事件、状态消息、工具日志和统一脱敏。

新增能力必须放在合适边界内。不得为了快速实现而绕过 `ToolRegistry`、脱敏、预算控制或引用校验。

## 开发流程

Codex 执行开发任务时应遵循：

1. 先读现有代码和测试，确认真实边界。
2. 找根因和结构性解法，避免只针对单个现象打补丁。
3. 先定义可验证目标，再改代码。
4. 改动保持窄而完整，避免无关重构。
5. 为新增行为补充测试，尤其是失败路径、安全约束和预算限制。
6. 运行相关测试；涉及共享流程、模型或 API 时运行完整后端测试。
7. 在最终回复中说明改了什么、验证了什么、是否有未完成风险。

## 质量门槛

以下情况不得视为完成：

- 只实现 happy path，没有失败路径测试。
- 代码绕过已有 registry、validator、budget、sanitize 或 citation 机制。
- LLM 输出未经过结构化解析就进入业务模型。
- 新增循环没有上限。
- 错误会泄露 API key、Bearer token、secret、token 等敏感信息。
- 报告引用无法被 validator 检查。
- 需要用户手动修正生成文件才能运行。

## 用户确认边界

Codex 可以自主决定常规实现细节，但以下情况必须向用户确认：

- 需求目标存在多种互斥解释。
- 需要删除或重写大量已有功能。
- 需要引入重量级依赖、外部服务、持久化系统或部署架构变化。
- 需要改变用户可见 API、数据模型、前端交互或配置格式。
- 完成目标需要牺牲安全性、测试覆盖或已有核心行为。

## 当前项目方向

近期开发重点按开发计划推进：

- 固化当前编排 state 和节点/服务约定。
- 实现外部 Skill 系统。
- 让总结、证据判断、补充搜索和报告生成逐步接入统一工具与状态机制。
- 保持后端完整测试稳定通过。
